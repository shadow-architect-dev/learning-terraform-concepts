# SRE 技術面接想定質問 ＆ 模範回答集 (Technical Interview Guide)

本ドキュメントは、本テンプレートを用いたポートフォリオを基に技術面談や採用面接に臨む際、面接官（特にテックリードや SRE）から突っ込まれやすい核心的な質問と、それに対する論理的かつプロフェッショナルな回答要点をまとめたチートシートです。

---

## 1. セキュリティ ＆ 暗号化に関する質問

### Q1. S3アセットバケットからの配信において、従来の OAI (Origin Access Identity) ではなく OAC (Origin Access Control) を採用した理由は何ですか？
* **面接官の意図**: AWSのセキュリティアップデートをキャッチアップしているか、またKMSとの親和性を理解しているか。
* **模範回答**:
  > 「AWSが現在OAIを非推奨とし、OACを推奨しているためです。技術的な決定打となったのは、**『KMSカスタムキーで暗号化された S3 バケットからの配信に対応している点』**です。
  > 本構成ではアセットバケットの暗号化に KMS CMK を適用しているため、旧来 of OAI では CloudFront から S3 へのアクセス時に KMS の暗号解読ができずエラーになります。OAC を採用し、かつ最新の SigV4 署名を用いることで、暗号化されたプライベートバケットからセキュアにキャッシュ配信を行う構成としています。」

### Q2. KMSで「AWS管理のデフォルトキー」ではなく「カスタマー管理キー (CMK)」をわざわざ作成して一元管理しているのはなぜですか？
* **面接官の意図**: セキュリティガバナンスと、キーマネジメント（鍵管理）の主導権の重要性を理解しているか。
* **模範回答**:
  > 「デフォルトの AWS 管理キーは手軽ですが、キーの利用制限（キーポリシーのカスタマイズ）や、インシデント発生時の即時無効化（一時的な鍵のロック）、削除のコントロールなどが一切行えません。
  > 自社専用の CMK を作成することで、キーポリシーを使い『この ECS タスクと、特定のログサービスのみが復号できる』という最小権限を鍵自体に仕込むことができます。
  > また、これを『一元管理』として、DB・シークレット・コンテナログ・監査ログのすべてのレイヤーに同一のキーを適用することで、管理オーバーヘッドと鍵の維持費を最小化しつつ、システム全体の多層暗号化（Defense in Depth）を実現しています。」

## 2. データベース ＆ シークレット管理に関する質問

### Q3. Secrets Manager の自動ローテーションにおいて、なぜ「マルチユーザー」ではなく「シングルユーザー (MYSQL_ROTATION_SINGLE_USER)」を選んだのですか？パスワード更新時にアプリの接続瞬断が発生しませんか？
* **面接官の意図**: ローテーションの仕組みと、一時的な認証不整合（ダウンタイム）への対策を理解しているか。
* **模範回答**:
  > 「シングルユーザー方式は管理者（マスター）パスワードを直接書き換えるため、通常はパスワード変更からコンテナへの反映までの間に接続エラー（瞬断）が起きるリスクがあります。
  > しかし、本テンプレートでは `stg` / `prod` 環境に **RDS Proxy** を導入しています。アプリコンテナは RDS Proxy と常時接続しており、DBへの認証とコネクション管理は RDS Proxy がすべて肩代わりしています。
  > パスワードがローテーションされた際も、RDS Proxy が Secrets Manager から最新の値をロードして Aurora との接続を自動追従させるため、アプリ側は再起動やエラーを起こすことなく、ゼロダウンタイムで透過的なローテーションが可能となっています。このため、追加ユーザーリソースの不要なシンプルかつセキュアなシングルユーザー構成を選択しました。」

### Q4. 開発環境 (dev) でデータベースやシークレットの自動ローテーションを「オフ」にしているのはなぜですか？
* **面接官の意図**: FinOps（クラウドコスト最適化）に対する現実的な判断能力があるか。
* **模範回答**:
  > 「環境パリティの維持は重要ですが、コストとのトレードオフを考慮した結果です。
  > Secrets Manager の自動ローテーションを有効化すると、ローテーション用の Lambda 関数が VPC 内に常時プロビジョニングされ、ローテーション実行時のAPI手数料も発生します。開発環境（dev）には機密データが存在せず、外部への公開も行わないため、セキュリティ要件（PCI-DSS等の監査要件）が必須となる `stg` / `prod` でのみ有効化し、`dev` ではスキップすることで無駄な運用コストを完全に削減しています。」

## 3. ネットワーク ＆ 可観測性 (Observability) に関する質問

### Q5. Datadog Agent (サイドカー) の起動を、アプリケーションコンテナよりも「先」に完了（START）させる依存関係を定義しているのはなぜですか？
* **面接官の意図**: サイドカーパターンの通信モデルと、起動時（初期化フェーズ）のトラブルシューティングの重要性を理解しているか。
* **模範回答**:
  > 「アプリケーションのライフサイクルにおいて、最もクラッシュや遅延が発生しやすいのは起動直後の数秒間（初期化やDB接続プール確立フェーズ）です。
  > アプリケーションから Datadog への通信は『プッシュ型（アプリ内のSDKが localhost:8126 に向けてトレースを能動的に送信する方式）』であるため、データ受信機である Datadog Agent が先にポートを開けて待機している必要があります。
  > もしアプリを先に起動させてしまうと、起動直後の決定的な初期化エラーや起動遅延のメトリクスが Datadog に届かず消失してしまうため、CDK で `ContainerDependency` を用いて起動順序を厳密に制御しています。」

### Q6. 開発環境（dev）で NAT Gateway の数を `0` に設定していますが、Fargateコンテナが ECR からイメージをプルしたり、Secrets Manager と通信したりすることはどうやって可能にしているのですか？
* **面接官の意図**: プライベートネットワークにおける通信経路設計（VPCエンドポイント）の知識があるか。
* **模範回答**:
  > 「検証モードやローカル開発時に NAT Gateway のプロビジョニング費用（月額数千円〜）を完全に回避するため、VPC 内に **VPC エンドポイント（AWS PrivateLink）** を構築し、ECR（APIおよびS3ゲートウェイ）、Secrets Manager とのプライベート通信経路を確保しています。
  > これにより、インターネット（NAT Gateway）を経由することなく、AWSのプライベートバックボーンを経由して安全かつ高速にイメージプルや資格情報のロードを完了させています。」

## 4. テスト自動化 ＆ ガバナンスに関する質問

### Q7. CDK assertions を用いたインフラ単体テスト (Jest) を作成する実務上のメリットは何ですか？
* **面接官の意図**: 単に『デプロイできないからテストしている』という消極的理由ではなく、IaC変更管理におけるテスト自動化の本質的価値を理解しているか。
* **模範回答**:
  > 「実務における最大メリットは、**『インフラの変更管理・リファクタリングにおける安全性の確保』**と**『セキュリティポリシーのコードによる強制（コンプライアンスの自動化）』**です。
  > CDK のコード整理や依存ライブラリの更新を行った際、意図しないリソースの削除・再作成が起きないかを、実際の AWS 環境にデプロイすることなくローカルや CI パイプライン上で数秒で検証できます。
  > また、『本番データベースは削除保護が有効であること』『すべてのロググループに KMS キーが指定されていること』などの組織のコンプライアンスルールをテストコードでアサーションしておくことで、構成ミスを含んだコードが本番へデプロイされるのを確実にブロックできます。」

## 5. 共通基盤・集約ネットワーク ＆ プラットフォームエンジニアリングに関する質問

### Q8. 各ワークロード（EKS/ECS）側の VPC から NAT Gateway を廃止し、共通基盤（Shared Services）へアウトバウンドを Transit Gateway (TGW) 経由で集約した（Common Egress）理由は何ですか？
* **面接官の意図**: クラウドコスト最適化（FinOps）およびセキュリティ統制におけるアーキテクチャのトレードオフを理解しているか。
* **模範回答**:
  > 「理由は**『劇的なコスト削減（FinOps）』**と**『セキュリティ統制の一元化』**の二重の効果があるためです。
  > 各ワークロードVPCごとに NAT Gateway をマルチAZ構成（各2〜3台）で建てる構成では、NAT GWの固定基本料金（1台あたり約4.5万円/月）が『アカウント数 × AZ数』分だけ乗算され、非常にコスト高になります。
  > 共通基盤（Shared Services）に NAT Gateway を集約し、TGW 経由でトラフィックを逃がすことで、個々のワークロードの NAT 基本料金を完全にゼロに圧縮できます。また、インターネットの出口が1箇所に集約されるため、将来的に AWS Network Firewall などの高機能なプロキシ・次世代ファイアウォールを1箇所に挟むだけで、組織全体のインターネット行き通信の監査と不正通信ブロックを一元化できるセキュリティ上のメリットがあります。」

### Q9. VPC CIDRブロックをハードコードせず、AWS VPC IPAM（IP Address Manager）から動的に切り出す構成（CDK / Terraform）を採用した理由は何ですか？
* **面接官の意図**: マルチアカウント環境におけるIPアドレスバースト管理と、衝突回避の自動化を理解しているか。
* **模範回答**:
  > 「将来的な閉域接続（TGW経由のVPCピアリング）時における**『IPアドレス衝突（CIDR競合）』を物理的に防ぐため**です。
  > 各リポジトリでCIDRを `10.0.0.0/16` などと静的にハードコードして構築していると、将来的にそれらの環境同士を相互接続しようとした際、IP範囲が衝突してルーティングが行えなくなります。
  > 共通基盤側で一元管理している IPAM プール（例: `10.0.0.0/8`）から動的に `/16` を切り出す設計にすることで、開発・検証・本番、またはEKSやECSなどのシステム種類を問わず、自動的に重複のないIP帯域でVPCがプロビジョニングされ、組織全体のクリーンな閉域ネットワーク統合が可能になります。」

### Q10. ポートフォリオ用のプレースホルダー値（`tgw-dummy-...` 等）を用いつつ、CDKのAssertionsテスト（単体テスト）を壊さずに動的切り替えを実装するために、コード上でどのような工夫をしましたか？
* **面接官の意図**: テスト容易性（Testability）と後方互換性を保ちながらインフラコードをリファクタリングする実務能力があるか。
* **模範回答**:
  > 「単に IPAM や TGW の接続IDパラメータを必須化するだけでは、モック無しでローカルで回す既存の単体テスト（Jest assertions）がパラメータエラーで動作しなくなったり、検証アサーションが全て失敗してしまいます。
  > そこで、`ipamPoolId` が引数として指定されていない場合は、従来の手動CIDR（`10.0.0.0/16`）および自前 NAT GW を立ち上げる構成へと自動的に**『フォールバック（後方互換モード）』**するよう、CDK/TypeScript のコードロジックを設計しました。
  > これにより、ローカルでの高速なテスト互換性を100%維持しながら、共通基盤からパラメータが注入された本番デプロイ時には、自動的にIPAMプールからの動的切り出しおよびTGW集約アウトバウンド（自前 NAT GW は `0` 台化）へと切り替わる柔軟なコード設計を実現しました。」

### Q11. Platform Engineering (Backstage) や GitOps (ArgoCD) などの思想をこの IaC ポートフォリオに組み込んだ狙いは何ですか？
* **面接官の意図**: クラウドインフラを単に構築する『インフラエンジニア』と、開発者の生産性を最大化する『プラットフォームエンジニア』の違いを認識しているか。
* **模範回答**:
  > 「クラウドインフラの構築自体は IaC で自動化できても、開発者がインフラのプロビジョニングやデプロイを毎回SREに依頼している状態では、組織全体のデリバリースピードがスケールしません。
  > そこで、インフラ構成を Backstage テンプレートを用いてパッケージ化してセルフサービス（Golden Path）提供し、さらにインフラ作成後のアプリケーションデプロイは ArgoCD による GitOps で自動同期させることで、開発者がインフラの専門知識や強い特権権限を意識せず、安全・高速にアプリケーションを本番環境へデプロイできる体制を目指しています。
  > 単に個別のサーバを構築するだけでなく、**『開発生産性を最大化し、認知負荷を下げるプラットフォーム』を組織向けに提供する**というプラットフォームエンジニアリングの考え方を意識して設計しています。」
