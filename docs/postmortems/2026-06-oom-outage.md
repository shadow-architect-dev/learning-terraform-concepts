# Post-Mortem: 2026-06-25 App Pod OOM Crash Loop due to Memory Leak

* **日付**: 2026年6月26日
* **著者**: SRE チーム (小川)
* **ステータス**: 完了 (Resolved)
* **インシデント種別**: P2

---

## 1. 概要 (Summary)

2026年6月25日 10:00 (JST) の定期リリースにおいて、画像処理モジュールにメモリリークを引き起こす致命的なコードが含まれた状態で本番デプロイが実行されました。これにより、特定の大きな画像に対するリクエストが発生した際に Pod 内のメモリ使用量が急激に高騰し、Kubernetes の OOMKilled が多発しました。

一時的に `CrashLoopBackOff` 状態に陥り、ALB から `502 Bad Gateway` が返却されました。最終的にリリースを 1 世代前のバージョンにロールバックすることで 10:25 に完全復旧しました。この障害により、30日間の可用性エラー予算の **5.2%** を消費しました。

## 2. ユーザーへの影響 (User Impact)

* **エラー率**: 影響時間中、全リクエストの **12.4%** が 502 / 504 エラーとなりました。

## 4. 根本原因分析 (5 Whys)

1. **なぜシステムの応答が止まったか？**
   * ➡ アプリケーションコンテナが頻繁に強制終了されていたから。
2. **なぜコンテナが強制終了されたか？**
   * ➡ Kubernetes の OOMKilled 機能が働き、メモリ制限を超えたコンテナが破棄されたから。
3. **なぜコンテナのメモリ制限を超えたか？**
   * ➡ アップデートで導入された新しい画像処理ライブラリが、画像リサイズ後にメモリ領域（C言語領域のバッファ）を開放せず、メモリリークを起こしていたから。
4. **なぜリリース前のテストでこのメモリリークが検知できなかったか？**
   * ➡ 検証環境（Staging）のテストデータが本番の数万点に及ぶ実データと比較してサイズが小さく、少数のテストではリーク量が制限内に収まってしまっていたため。
5. **なぜ検証環境のデータと負荷が不足していたか？**
   * ➡ リリース前の自動負荷テスト（Load Testing）ステージで、本番の多様なアセットサイズを再現するデータセットの同期が行われていなかったため。
